08 — Litaniae (Packages)

A litania is a versioned, distributable package of modules — Vox Gothica's answer to Python packages and Go modules. The design is deliberately Go-shaped: git-first, decentralized, checksum-locked, minimal version selection. No central server is required to ship v0.2; a curated registry (the Librarium Sanctum) layers on top later without changing the model.

1. Identity

A litania is identified by its via (import path):


<auctor>/<nomen>                  ++ short form, e.g.  adeptus/http_sancta
<host>/<auctor>/<nomen>           ++ full form, e.g.  github.com/adeptus/http_sancta
  • auctor and nomen match [a-z_][a-z0-9_-]* (hyphens allowed in package names, not in module files).
  • The short form resolves through the Librarium index (§7); the full form resolves directly via git. The full form always works even if the Librarium burns — decentralization is doctrine.
  • Reserved auctor names: fons, litaniae, gothica, and all stdlib module names (X-IV on publish attempt).

2. Versioning (normative)

  • Semantic versioning, Arabic: MAJOR.MINOR.PATCH (git tags v1.2.3). *(Roman versions were considered and rejected: v.XLII.IX.III is illegible in a diff. The Machine God favors legible diffs.)*
  • Pre-1.0: anything may break. Post-1.0: breaking changes require a MAJOR bump — violating this earns your version an excommunication entry in the Librarium (X-VI).
  • Major versions are distinct litaniae, Go-style: adeptus/http_sancta/v2 is imported and resolved independently of v1. A program may depend on both. This kills the diamond-dependency problem at the cost of an ugly suffix — a trade the Adeptus accepts.

3. The manifest — litania.toml

Every litania (and every project) carries one at its root:


[litania]
via         = "github.com/adeptus/http_sancta"
versio      = "1.4.2"                  # required for published litaniae
descriptio  = "Sanctified HTTP rites for the discerning magos"
auctores    = ["Magos Errant Kane <kane@mechanicus.example>"]
licentia    = "MIT"
gothica     = ">=0.2"                  # minimum toolchain version

[requirit]                             # dependencies
"github.com/adeptus/codex_json"  = "^1.2"
"github.com/xenos/retis/v2"      = "^2.0.1"

[requirit_probationis]                 # test-only dependencies
"github.com/adeptus/probatio_aux" = "^0.3"

[canticum]                             # only for runnable projects
principium = "principium.vg"           # entry file for `gothica invoco` with no args

Constraint syntax: ^1.2 (≥1.2.0, <2.0.0), ~1.2.3 (≥1.2.3, <1.3.0), =1.2.3 (exact), >=1.2 (at least). Malformed manifest: X-IV manifestum_pravum.

4. Version resolution — Minimal Version Selection (normative)

Given the dependency graph, the selected version of each litania is the maximum of the minimum versions requested by all requirers — Go's MVS, verbatim:

  1. Collect every constraint on litania L across the graph.
  2. Each constraint has an implied minimum; take the highest minimum, v.
  3. If v violates any upper bound (^/~/=): X-II versio_discors, message lists the conflicting requirers.
  4. Otherwise select exactly v — never silently the newest available.

*(Rationale: builds are reproducible from the manifest alone, no SAT solver, upgrades are always explicit acts of faith — gothica renovo.)*

5. The lockfile — litania.claustrum

Machine-written, human-readable, committed to version control:


[[claustrum]]
via     = "github.com/adeptus/codex_json"
versio  = "1.3.0"
fons    = "git+https://github.com/adeptus/codex_json@v1.3.0"
signum  = "sha256:9f2c1a...e77b"       # hash of the package archive tree
  • signum is verified on every fetch and every build; mismatch is X-III signum_falsum and there is no flag to override it. A tampered litania is not installed with a warning; it is not installed.
  • Lockfile inconsistent with manifest: X-V, fix with gothica adfero.

6. Installation layout

gothica adfero materializes dependencies into the project-local litaniae/ directory (vendoring is the default and only mode in v0.2 — a global content-addressed cache ~/.gothica/librarium/ seeds it, so disk cost is one hardlink):


litaniae/
  github.com/adeptus/codex_json/     ++ the package's fons/ tree + its litania.toml
  github.com/xenos/retis/v2/

Import resolution (07-modules §3) searches this tree. Nested litaniae/ inside dependencies are ignored — the root project's lockfile governs the whole build (single flat resolution, no node_modules hell).

7. The Librarium (registry)

  • v0.2: the Librarium is just a git repository of index files mapping short vias to full vias + advisory metadata (excommunicated versions, license, description). Default: github.com/vox-gothica/librarium-sanctum; override or add mirrors in ~/.gothica/configuratio.toml (first match wins — private corporate librariums are first-class).
  • Publishing (§8) into the Librarium = a pull request adding/updating your index entry, exactly like early Homebrew. A web UI and API are M6 concerns; the data model already supports them.
  • Excommunication ≠ deletion. A yanked (excommunicata) version stays fetchable if already in a lockfile (builds must not break) but is refused for new resolution, with the advisory shown.

8. Package commands

CommandAct
gothica initiumscaffold a project (litania.toml, principium.vg, fons/, probationes/)
gothica initium --litaniascaffold a publishable litania
gothica adferoresolve manifest → update lockfile → populate litaniae/
gothica adfero <via>[@versio]add a dependency (writes [requirit], then as above)
gothica expello <via>remove a dependency; fails if still required transitively
gothica renovo [<via>]raise constraint minimums to newest compatible; explicit upgrades only
gothica offeropublish: verify manifest, run gothica proba, check tag == versio, verify no path deps, push tag; prints the Librarium PR link
gothica librarium quaere <term>search the index
gothica librarium inspice <via>show versions, advisories, deps of a litania

offero refuses (X-IV) if: tests fail, the git tree is dirty, versio ≠ the git tag, [requirit] contains a path or pre-release dependency, or via doesn't match the repo origin.

9. What a litania may contain (normative)

  • Only LITANIA-mode .vg files, litania.toml, docs, and probationes/. A CANTICUM/FABRICA inside a published litania is refused at publish and ignored at import (packages are libraries; ship apps as repos, not litaniae).
  • FABRICA helper litaniae are legal and encouraged: LITANIA modules exporting rites that call EXSTRUATUR. Such rites execute only when invoked from a FABRICA context; invoked from CANTICUM they raise fabrica_in_cantico. This is how the ecosystem grows reusable infrastructure patterns ("a blessed VPC", "a consecrated Postgres") — the Vox Gothica equivalent of Terraform modules, but they are just code.
  • Native/FFI extensions: not in v0.2 (LEGATUS is a reserved keyword for this future).

10. Local development

Path override for developing two litaniae in tandem — in the project's manifest only:


[substitutio]
"github.com/adeptus/codex_json" = { via_localis = "../codex_json" }

Substitutions are ignored by offero targets and rejected in published manifests (X-IV). Equivalent of Go's replace.

11. Naming taste (informative)

Litania names should be Latin-flavored and honest: codex_json, http_sancta, tempus_extra, postgres_benedicta. The Librarium index review rejects names that collide with stdlib or claim false sanctity (officialis_* is reserved for the core team).
