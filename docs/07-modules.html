07 — Modules (in-project)

A module is one .vg file. This chapter covers modules within a single project; packages of modules shared between projects are litaniae (08-litaniae).

1. Module kinds

  • LITANIA "name". header — a library module: only declarations (DECLARO/SANCTUM/RITUS/SCHEMA and imports) at top level. Imperative top-level statements are PROFANATIO P-XVI. (Rationale: importing must be effect-free so import order can never matter.)
  • CANTICUM / FABRICA files may also be imported? No. Importing a CANTICUM or FABRICA is P-XVI. Entry files and library files are different castes.

2. Import forms


INVOCO "mathematica"                       ++ stdlib module
INVOCO "fons/logistics"                    ++ project module (path from project root, no .vg)
INVOCO "adeptus/http_sancta"               ++ installed litania (08-litaniae)
INVOCO "adeptus/http_sancta" UT rete       ++ alias
INVOCO radix, potentia EX "mathematica"    ++ named imports, bound directly
  • Default binding name is the final path segment (logistics, http_sancta). Members are accessed as logistics.expedi(...).
  • EX binds listed names directly; importing a name that is not exported raises P-XVII nomen_non_exportatum at load time.
  • Import statements must precede all other declarations in the file (after the header). Elsewhere: P-XVI.

3. Resolution order (normative)

For INVOCO "path":

  1. Stdlib names (no / in path, matches a stdlib module) — always win; shadowing stdlib is impossible.
  2. Project modules: <project root>/<path>.vg, where project root is the directory containing litania.toml (or the entry file's directory if no manifest).
  3. Installed litaniae: <root>/litaniae/<auctor>/<nomen>/... (see 08-litaniae §6).
  4. Otherwise: load-time P-XVIII invocatio_vana ("the invocation went unanswered"), with a hint listing near-miss names.

4. Semantics

  • A module is loaded once per process; subsequent imports bind the same instance (module-level mutable state is therefore shared — use sparingly; lustro warns on mutable DECLARO at module top level).
  • Loading executes the module's declarations top to bottom. Rite names are hoisted within the module (05-rites §5); cross-module references are not hoisted — import order in a file is the initialization order.
  • Import cycles raise circulus_impius (Major) naming the full cycle path.

5. Visibility

  • Everything at module top level is exported unless its name starts with _ (unhallowed = private).
  • PALAM ("publicly") before a declaration is redundant but legal, for emphasis: PALAM RITUS .... lustro --strictus can require it.
  • Schema fields are always visible; there is no field privacy in v0.2.
